Claude Codeの使い方

あなたは高度に委任志向のワークフローを持ち、Claude Codeをコーディングアシスタントとgit自動化レイヤーの両方として活用しています。セッションのかなりの部分 — 約3分の1 — がgit操作に費やされています：日本語Conventional Commitメッセージによるスマートコミット、論理的なdiff分割、プッシュ、PR作成など。勤怠PDF処理やスマートコミットなどのタスクに対して再現可能なパターン（おそらくカスタムコマンド）を構築し、Claudeを会話相手ではなく自動パイプライン実行エンジンとして扱っています。殺人ミステリーメモアプリのような機能開発に取り組む際は、詳細な事前仕様や構造化されたプランを提供し、Claudeに実装を任せる傾向があります。TodoWriteが59回、ExitPlanModeが12回使用されていることから、作業前にプランモードで整理する頻度が高いことが確認できます。

とはいえ、あなたは受動的な委任者ではありません — Claudeが逸脱した際には断固として介入しています。Claudeが過剰設計のuseSyncExternalStoreアプローチを提案した際にシンプルさへとリダイレクトし、ツール選択を複数回修正し（npm→bun、npx→bunx）、不正確な技術説明を指摘し、「段階」や「瞬間」といった不自然な日本語表現にも異議を唱えました。摩擦データには「間違ったアプローチ」が9回、「バグのあるコード」が7回記録されていますが、成果は強力に維持されています（完全達成23件、部分達成2件のみ）。これは、セッションを放棄するのではなく、素早く軌道修正する能力の高さを示しています。メモアプリとMarpプレゼンテーションの反復デザイン改善セッションでは、ビジュアル出力を確認し、的確なフィードバックを提供し、Claudeに過度な計画なしで調整を実行させるパターンが見られます。

使用言語（Markdownが83、TypeScriptが64と多い）と、125件のノートを永続ノートに統合したりNotionページからPDFサマリーを生成したりするタスクを組み合わせると、あなたは従来の開発に加えてナレッジ管理とコンテンツ作成にもClaude Codeを使用していることがわかります。短く集中的なバーストで作業しています — わずか5時間で34セッション・221メッセージは、セッション平均が10分未満であることを意味します。これはClaude Codeを長時間のペアプログラミングではなく、高速タスク完了のためのパワーツールとして扱うユーザーの特徴です。

主要パターン：あなたはClaude Codeを、事前定義されたワークフローによる高速タスク実行ツールとして運用し、間違ったアプローチや不正確なツール使用があった場合にのみ、鋭く的確に介入します。

問題が起きる場面

主な摩擦ポイントは、Claudeが間違ったツールや過剰設計のアプローチをデフォルトで選択してしまうことに起因し、繰り返しの修正と中断を余儀なくされています。

パッケージマネージャーとツールのデフォルト設定の誤り
Claudeはbunベースのプロジェクトでnpm/npxをデフォルトで使用してしまい、中断して修正する必要が繰り返し発生しました。CLAUDE.mdですべてのプロジェクトでbunを使用するよう指定すれば、Claudeがnpmにフォールバックすることを防げます。
  • Claudeが依存関係操作にnpmコマンドを使用し、ロックファイルの動作について不正確な説明をしたため、セッション中に修正が必要になった
  • Claudeがnpmベースのpdfエクスポートを試みてキャッシュ権限の問題で少なくとも2回失敗し、毎回bunxの使用を提案する必要があった
過剰設計と直接行動の代わりの過度な計画
Claudeは不必要に複雑なソリューションを提案したり、素直な実装を求めている場面で繰り返し計画サイクルに入ることがあります。「最もシンプルなアプローチを使って」「計画をスキップして、そのまま実装して」といった明示的なシンプルさの制約をプロンプトに前置きすることで対処できます。
  • Claudeがシンプルな直接読み取りパターンで十分なlocalStorageリファクタリングにuseSyncExternalStoreを提案し、中断してよりシンプルなアプローチを求める必要があった
  • プレゼンテーションのデザイン改善を依頼した際、Claudeが改善を直接適用する代わりに複数の計画イテレーションと質問を繰り返し、進捗が停滞した
即座の修正を要するバグのあるコード変更
Claudeの編集 — 特に複数ファイルや大規模な置換 — は構文エラー、重複コード、破損を頻繁に引き起こし、発見して修正を依頼する必要があります。編集後に結果を提示する前に常に型チェックとlintを実行するよう指示すれば、これらの問題を事前にキャッチできます。
  • メモアプリへのClaudeの初期デザイン変更が4つのコンポーネント（MemoPage、MemoInput、ResetButton、MemoItem）にわたって重複コードと構文エラーを引き起こし、500エラーが発生。中断してスクリーンショット前の事前チェックを要求する必要があった
  • コード置換がファイル末尾の重複コードにより構文エラーを引き起こし、編集直後にリンターを実行していれば回避できた追加の修正ステップが必要になった

試すべきClaude Code機能

CLAUDE.mdへの追加候補

Claude Codeにコピー＆ペーストするだけでCLAUDE.mdに追加されます。

複数のセッションでClaudeがnpm/npxをデフォルトで使用し、毎回ユーザーの修正が必要になった — 繰り返し発生する摩擦ポイント。
少なくとも8つのセッションで日本語Conventional Commitによるスマートコミットワークフローが使用された — これはコード化すべき中核的な反復ワークフロー。
複数のセッションでClaudeがUI変更時に構文エラーや重複コードを導入し、500エラーが発生。ユーザーが中断して事前チェックを要求する必要があった。
複数のセッションでClaudeがソリューションを過剰設計し（useSyncExternalStore、過度な計画サイクル）、ユーザーがシンプルなアプローチを求めて中断する必要があった。
プレゼンテーションデザインのセッションで、Claudeが改善を直接適用するのではなく計画サイクルを繰り返し、ユーザーを苛立たせた。

Claude Codeにコピー＆ペーストするだけでセットアップが始まります。

カスタムスキル
単一の/コマンドで実行できる再利用可能なプロンプト
あなたにとっての利点：すでに8以上のセッションで使用しているスマートコミットワークフロー — diffの分析、変更のグループ化、日本語Conventional Commitメッセージの作成、プッシュ — があります。/commitスキルを作れば、毎回これらの指示を繰り返す必要がなくなります。
mkdir -p .claude/skills/commit && cat > .claude/skills/commit/SKILL.md << 'EOF' ## スマートコミット 1. `git diff --staged`（ステージングなしの場合は`git diff`）を実行して変更を分析 2. 関連する変更を論理的なコミットにグループ化 3. 日本語でConventional Commitメッセージを作成 4. 提案されたコミットを提示し、ユーザーの承認を待つ 5. コミットを実行しリモートにプッシュ ルール： - `git`コマンドを直接使用（npm/npxは使わない） - 常にConventional Commit形式を使用：feat:, fix:, chore:, docs:, refactor: - コミットメッセージは日本語で記述 - プッシュ前にユーザーに確認 EOF
フック
特定のライフサイクルイベントに応じて自動実行されるシェルコマンド
あなたにとっての利点：複数のセッションでClaudeが結果を提示する前に構文エラーや壊れたコードを導入しました。型チェックを実行するプリエディットフックがあれば、7回発生した「バグのあるコード」の摩擦を自動的に防止できます。
# .claude/settings.json に追加： { "hooks": { "postToolUse": [ { "matcher": "Write|Edit", "command": "ext=$(echo $CLAUDE_FILE_PATH | sed 's/.*\\.//'); if [ \"$ext\" = \"ts\" ] || [ \"$ext\" = \"tsx\" ]; then bunx tsc --noEmit 2>&1 | head -20; fi" } ] } }
カスタムスキル
よく使うブランチ+コミット+PRワークフローの/prスキルを作成
あなたにとっての利点：少なくとも3つのセッションでコミット後にPRを作成しています。/prスキルを作れば、スマートコミットロジックとブランチ作成、PR送信を1つのコマンドにまとめられます。
mkdir -p .claude/skills/pr && cat > .claude/skills/pr/SKILL.md << 'EOF' ## PR作成 1. 現在のブランチを確認（mainの場合は新しいブランチを作成） 2. /commitスキルを実行して変更を整理・コミット 3. ブランチをリモートにプッシュ 4. わかりやすいタイトルと説明でPRを作成 ルール： - PRタイトルは英語、Conventional Commitスタイル - PR本文は箇条書きで変更を要約 - PR作成には`gh pr create`を使用 - 常に`bun`/`bunx`を使用、npm/npxは使わない EOF

Claude Codeの新しい活用方法

Claude Codeにコピー＆ペーストするだけでガイドが始まります。

スマートコミットワークフローをコード化する
最も頻繁なタスク（8セッション以上）はdiffの分析と日本語Conventional Commitの作成。これを1コマンド操作にしましょう。
Claude Codeセッションのかなりの部分をgit操作に費やしています — スマートコミット、プッシュ、PR作成がトップゴールの約半分を占めています。毎回同じ要件を説明しています：Conventional Commit形式、日本語メッセージ、論理的なグループ化。/commitカスタムスキルを作成すれば、すでに好みを知っている1つのコマンドに変わります。CLAUDE.mdのgitコンベンション項目と組み合わせれば、Claudeに2度説明する必要がなくなります。
Claude Codeに貼り付け：
git diffを分析し、日本語メッセージで論理的なConventional Commitにグループ化し、承認を待ってからコミット・プッシュする、.claude/skills/commit/SKILL.mdにカスタムスキルを作成してください。
壊れたコードに対するガードレールを追加する
最大の摩擦源はバグのあるコード（7回）と間違ったアプローチ（9回）。編集後の自動型チェックでほとんどをキャッチできます。
特にUI開発セッションで、Claudeが構文エラー、重複コードブロック、壊れたコンポーネントを複数回導入しました。Claudeが次のステップに進む前に確認するよう中断して依頼する必要がありました。Write/Edit操作のたびにTypeScript型チェックを実行するpostToolUseフックがあれば、エラーを即座に表面化させ、Claudeが次のステップに進む前にキャッチできます。これにより、リアクティブな「破損を修正する」ループが、プロアクティブな「即座にキャッチする」ワークフローに変わり、最も一般的な不満から解放されます。
Claude Codeに貼り付け：
TypeScriptファイルへのWriteまたはEditの後にbunx tsc --noEmitを実行するClaude Codeフックを.claude/settings.jsonに設定して、型エラーを即座にキャッチできるようにしてください。
中断を減らすためにプロジェクト規約を前置きする
多くの摩擦ポイントは、Claudeがプロジェクト規約を知らないことに起因しています（npmよりbun、複雑よりシンプル、日本語の好み）。
34セッションにわたり、同じことを繰り返し修正しました：npmではなくbunを使って、過剰設計しないで、日本語でコミットメッセージを書いて、UIを見せる前にコードをチェックして。これらの修正は時間を浪費し、フローを中断させます。これらの規約を記載した構造化されたCLAUDE.mdがあれば、Claudeはセッション開始時にそれを読み、自動的に従います。これは特にあなたにとって効果的です。なぜなら、34という高いセッション数と比較的短い平均セッション時間を持つため、2〜3回の修正後ではなく、最初のメッセージから正しく動作する必要があるからです。
Claude Codeに貼り付け：
このプロジェクトの包括的なCLAUDE.mdを作成してください。主な規約：常にbun/bunxを使用（npm/npxは使わない）、日本語のConventional Commitメッセージ、抽象化よりシンプルなソリューションを優先、UI変更を提示する前に必ず型チェックを実行、確認質問を繰り返す代わりにデザイン変更を直接適用。

フルスタック機能開発のための並列エージェント
すでに印象的な複数ファイル変更（5件の成功事例）とフルアプリスキャフォールディングを行っていますが、各セッションは逐次的です。並列サブエージェントを使えば、UIコンポーネント、APIルート、テスト、ドキュメントを同時に生成し、結果をマージできます。計画から仕上げまで複数セッションかかった殺人ミステリーメモアプリも、1回の協調的なバーストで構築できるかもしれません。
始め方：Claude CodeのTaskツールまたはSDK経由で複数のClaude Codeインスタンスをオーケストレーションして並列ワークストリームを実行しましょう。1つのエージェントがコンポーネント、もう1つがテスト、3つ目がスタイリングを担当 — すべて共有プランドキュメントから作業します。
Claude Codeに貼り付け：
オーケストレーターとして機能してください。この機能を並列ワークストリームに分割し、Taskツールを使ってサブエージェントをディスパッチしてください。ワークストリーム1：TypeScript型付きReactコンポーネントの作成。ワークストリーム2：各コンポーネントの包括的なVitestテストの作成。ワークストリーム3：このデザイン仕様に合わせたCSS/スタイリングの実装。各サブエージェントは独立して作業し、その後あなたが結果をマージし、競合を解決し、グリーンになるまでフルテストスイートを実行してください。機能仕様：[機能の説明]
スマートバリデーション付きゼロ介入Gitワークフロー
Git操作はあなたの最も頻繁なユースケースです（コミット、プッシュ、PRで13以上のセッション）。スマートコミットコマンドも構築済みですが、コミット探索中のユーザー中断やパッケージマネージャーの誤り（npm vs bun）による摩擦がまだ残っています。完全自律パイプラインなら、diffの分析、論理的なコミット分割、プレコミットチェックの実行、日本語Conventional Commitメッセージの生成、プッシュ、説明付きPRの作成を、ルーティン変更では承認ステップなしで実行できます。
始め方：既存のスマートコミットスラッシュコマンドを、Claude Codeフックとnpmよりbunを強制しコミット規約を定義するCLAUDE.md命令を使って、フルCI的パイプラインに拡張しましょう。
Claude Codeに貼り付け：
このリポジトリのステージ済みおよび未ステージの変更をすべて分析してください。日本語メッセージのConventional Commits形式に従って論理的なコミットにグループ化してください。各提案コミットに対して：1) 関連ファイルのみをステージ、2) ステージされたファイルに対して`bun run typecheck`と`bun run lint`を実行、3) チェックが失敗した場合は問題を修正して再ステージ、4) 日本語のConventional Commitメッセージでコミット。すべてのコミットが作成されたら、現在のブランチにプッシュ。mainの場合はまずフィーチャーブランチを作成。その後、すべてのコミットを要約するマークダウン説明付きのPRを作成。CLIツールには`bunx`を使用し、`npx`や`npm`は使わないこと。承認を求めず、パイプライン全体を実行すること。
